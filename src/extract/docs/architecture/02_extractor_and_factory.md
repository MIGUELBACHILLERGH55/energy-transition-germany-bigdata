Decides which extractor subclass to instantiate 
(BatchExtractor or StreamingExtractor) based on the source mode.
Injects the proper strategies for download, parsing, and storage or streaming ^S8YJfDWZ

Input:
SourceSpec (validated and ready for execution)

Output:
Instance of either BatchExtractor or StreamingExtractor ^UsHGTtKY

Contract:
  - create(source_spec)
      → selects the correct extractor subclass based on:
          • mode: "batch" or "stream"
          • kind: "api", "file", "ftp", "db", "kafka", "ws", "mqtt", etc.

  - Injects the following dependencies automatically:
      • DownloaderStrategy → HttpDownloader | FileDownloader | FtpDownloader | DbDownloader
      • ParserStrategy     → CsvParser | JsonParser | ParquetParser
      • StorageClient      → FsClient | S3Client | DbClient

BatchExtractor → Purpose & Contract ^ieYjmVpk

 Orchestrates the complete batch extraction lifecycle — from planning
  and fetching to parsing and persisting data. Implements the Template
  Method pattern to define the execution flow, while delegating low-level
  operations to injected strategies. ^WBVNiQnK

Input:
  BatchSourceSpec (validated, resolved)
  DatasetSpecs (included within the source)

Output:
 Parsed and persisted batch data (clean, ready for downstream usage) ^FdQqshKM

Contract:
  - run()
      → orchestrates the entire lifecycle in order:
          1. prepare()        → setup headers, paths, timezone
          2. plan()           → build PlanItems
          3. fetch()          → download raw data using DownloaderStrategy
          4. parse()          → convert RawResponse → ParsedBatch via ParserStrategy
          5. persist_landing()→ write clean data to landing via StorageClient
          6. report()         → summarize logs and results

  - Injected strategies:
      • DownloaderStrategy → Http | File | Ftp | Db
      • ParserStrategy     → Csv | Json | Parquet
      • StorageClient      → Fs | S3 | Db

  - Guarantees:
      • Consistent and idempotent batch runs.
      • Clear separation of orchestration (this class) and execution (strategies).
      • No direct network or I/O logic inside the extractor itself ^G56nLXG2

StreamingExtractor → Purpose & Contract ^H7RCZtzA

  Manages continuous, incremental data ingestion from streaming sources
  (e.g., Kafka, WebSocket, MQTT). Handles message consumption, parsing,
  checkpointing, and output persistence without full restarts.
 ^usdPupRJ

Input:
   StreamingSourceSpec (validated, resolved)
  Stream configuration (topics, endpoints, windowing, checkpoints)

Output:
 Parsed streaming records (processed in real-time or micro-batches) ^BuL69kfG

Contract:
  - run()
      → orchestrates continuous streaming flow:
          1. connect()        → establish stream connection via StreamingClient
          2. consume()        → receive and buffer incoming messages
          3. parse()          → transform messages to structured records via ParserStrategy
          4. checkpoint()     → persist offset/state via CheckpointManager
          5. persist_landing()→ write processed data via StorageClient
          6. monitor()        → handle reconnections, metrics, alerts

  - Injected strategies:
      • StreamingClient    → Kafka | WebSocket | MQTT
      • ParserStrategy     → Json | Avro | Parquet
      • StorageClient      → Fs | S3 | Db
      • CheckpointManager  → Local | S3 | Redis

  - Guarantees:
      • At-least-once delivery with checkpoint recovery.
      • Stateless orchestration — externalized offsets & commits.
      • Seamless recovery from transient network or broker failures. ^AUObOKZJ

DownloaderStrategy → Purpose & Contract

Purpose:
  Encapsulates how raw data is fetched from a specific source type
  (API, file system, FTP, database, etc.). Handles retries, timeouts,
  and session management.

Input:
  PlanItem (contains dataset, endpoint/path/query, params)

Output:
  RawResponse (payload bytes or text, plus metadata)

Contract:
  - open_session()
  - fetch(plan_item) → RawResponse
  - close()

Implementations:
  HttpDownloader | FileDownloader | FtpDownloader | DbDownloader

Guarantees:
  • No parsing or persistence — retrieval only.
  • Supports retry/backoff and timeout policies.
  • Isolated, stateless between runs.
------------------------------------------------------------
 ^ta311P8h

ParserStrategy → Purpose & Contract

Purpose:
  Transforms a raw payload into a structured, minimal clean dataset.
  Applies schema validation and lightweight normalization.

Input:
  RawResponse (text/bytes) and PlanItem (format hints, schema info)

Output:
  ParsedBatch (records | DataFrame | bytes)

Contract:
  - parse(raw_response, plan_item) → ParsedBatch

Implementations:
  CsvParser | JsonParser | ParquetParser

Guarantees:
  • No I/O or persistence — pure transformation.
  • Must preserve partitioning and metadata.
  • Schema validation is optional but encouraged.

 ^v0I2TE8y

StorageClient → Purpose & Contract

Purpose:
  Persists parsed data into the physical storage defined by the
  current environment profile (FS, S3, or DB). Ensures idempotent
  writes and atomic partition handling.

Input:
  ParsedBatch (clean data) and resolved destination path/profile

Output:
  None (side-effect: data written to target storage)

Contract:
  - ensure_path(path)
  - write_batch(parsed_batch, path, mode)

Implementations:
  FsStorageClient | S3StorageClient | DbStorageClient

Guarantees:
  • Handles partitioning and file naming.
  • Idempotent per partition or write transaction.
  • Abstracts backend-specific I/O details.
